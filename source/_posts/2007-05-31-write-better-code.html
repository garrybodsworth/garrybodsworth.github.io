There is an interesting talk on Slashdot about coding here. It's good to see that there is still occasionally some discussions about programming on there.

The article is about Six Ways To Write More Comprehensible Code. It makes size points:
  1. Comment well and often.
  2. Use sensible names not obscure constants.
  3. Use variables with descriptive names.
  4. Do error checking.
  5. "Premature optimization is the root of all evil." - Donald Knuth
  6. Don't be too clever.


The article is a nice little bit about writing better code. Personally I rate number 5 very highly, in fact I should write an article about writing better code from my point of view.

ditto that on number 5.

"comment well" is a bit of a tautology. It's like saying:

"Rule 1 for writing good code is to write good code!"

I've heard it said that comments are a crutch and a sign that you aren't applying rules 2 and 3 properly. Often we stick comments above distinct chunks or paragraphs of code. This style recommends you split each paragraph into a little function of its own with a descriptive name.

I also want it to be understood that not all variables are the same. Often variables are structural -- created on one line and passed as an argument on the next. It's a distraction to give them descriptive names. They act as pronouns. Like using "she" instead of repeating the name in the next sentence.
